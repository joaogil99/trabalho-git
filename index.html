O commit tem a função de consolidar as alterações realizadas em um arquivo do seu projeto. Desta maneira, o estado atual do projeto é capturado e gravado no seu repositório local, registrando quais arquivos foram alterados e ainda quais partes desses arquivos sofreram mudanças, sendo assim, é possível manter um histórico de todas as fases e alterações realizadas no projeto, mantendo o controle sobre o seu versionamento. A melhor maneira de se utilizar o commit é com uma mensagem de commit de acordo com um Changelog, que irá servir para identificar quais foram as mudanças ocorridas nos arquivos do projeto e com qual finalidade elas foram realizadas.

Como dito na pergunta anterior, o commit tem a função de registrar as mudanças no repositório local, gravando o estado do projeto naquele momento, já o push tem a função de enviar essas informações para o repositório remoto, fazendo a conexão entre os dois projetos, para que então fiquem sincronizados um com o outro. Funciona da seguinte maneira: após realizar as mudanças desejadas nos arquivos de seu repositório local, utiliza-se o git commit para gravar as alterações, como se estivesse "salvando", após isto, o git push é utilizado para enviar essas mudanças para o repositório remoto, localizado no GitHub, por exemplo, e assim os dois repositórios ficam iguais.

Tag é uma maneira de se indicar um ponto específico na linha do tempo do projeto. Devem seguir as regras do versionamento semântico, em ordem crescente e tendo a sintaxe x.y.z. As tags são utilizadas para identificar os releases, que são os pontos importantes no projeto, seja pela implementação de uma nova funcionalidade, correção de bug ou melhoria de desempenho em uma função já existente. Ao se utilizar uma tag, indica-se que aquela parte do código foi "finalizada", devendo manter-se a sua integridade, caso ocorram outras mudanças, deve-se utilizar uma nova tag para indicar uma nova versão do projeto.

Branchs podem ser entendidas como uma ramificação do código fonte, ou uma nova linha do tempo, do projeto principal. São ramos secundários do projeto principal. As branchs são muito utilizadas em projetos onde há mais de um colaborador, de maneira que cada integrante da equipe possa trabalhar em sua branch separadamente, sem afetar o código da branch principal, a main. Após finalizar as funcionalidades ou correções desejadas em sua branch secundária, o desenvolvedor pode solicitar que a sua branch seja unida à principal através de um pull request. Depois da unificação, cria-se uma nova branch para as próximas alterações e o processo se repete.

Durante um projeto onde se utiliza ferramentas de controle de versionamento, pode ocorrer uma situação em que dois integrantes da equipe desejam alterar o mesmo arquivo, o que pode resultar em um conflito no git. Para solucionar este problema, deve ser realizada uma interferência manual diretamente no trecho conflitante, removendo os trechos conflitantes e realizando mais alterações, caso for necessário, Para evitar esse tipo de situação, é aconselhável que todos os integrantes da equipe estejam atualizados com a ramificação principal através do git pull, e que a equipe procure designar tarefas de forma que duas pessoas não precisem trabalhar no mesmo arquivo ao mesmo tempo.

Em caso de incêndio, git commit e git push". Este "ditado" existe para reforçar que deve-se salvar o estado do trabalho constantemente, para evitar que mudanças no código sejam perdidas em algum acidente que venha a danificar a máquina física ou até mesmmo por falha humana.
